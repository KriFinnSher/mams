# User Stories MAMS v1

## US-01. Вход в систему

**Как** авторизованный пользователь  
**Я хочу** войти в систему по выданным логину и паролю  
**Чтобы** получить доступ к сервисам своей организации

**Критерии приёмки**

- На стартовой странице есть только login-форма.
- При корректных credentials пользователь успешно входит в систему.
- После входа пользователь получает JWT-сессию.
- При некорректных данных отображается ошибка без раскрытия лишних деталей.
- После входа пользователь перенаправляется в интерфейс MAMS.

## US-02. Профиль пользователя

**Как** авторизованный пользователь  
**Я хочу** видеть свой профиль  
**Чтобы** понимать, в какой организации я работаю и как в системе интерпретируются мои права доступа

**Критерии приёмки**

- В профиле отображаются login и организация.
- В профиле нет одного глобального поля `role`.
- Пользователь видит пояснение, что его права определяются на уровне конкретного сервиса.
- В профиле отображается список сервисов, где роль пользователя отличается от `Observer`.
- Для каждого такого сервиса рядом отображается конкретная роль пользователя.
- Пользователь не может менять эти данные через UI первой версии.
- Доступ к профилю есть у всех авторизованных пользователей.

## US-03. Добавление сервиса в систему

**Как** любой авторизованный пользователь  
**Я хочу** добавить новый сервис через UI-форму  
**Чтобы** MAMS начал с ним работать как с отдельной сущностью

**Критерии приёмки**

- В системе есть отдельная страница/форма добавления сервиса.
- В форме можно указать минимум: название, описание, тип, репозиторий, основную ветку, важность, флаг PII, команду и идентификатор Grafana dashboard.
- После сохранения сервис появляется в списке сервисов организации.
- Данные сервиса сохраняются в PostgreSQL.
- Пользователь, создавший сервис, автоматически получает для него роль `Service Owner`.
- У сервиса может быть только один владелец одновременно.
- Остальные пользователи той же организации по умолчанию получают к новому сервису доступ уровня `Observer`, пока им явно не назначена роль `Developer`.

## US-04. Просмотр карточки сервиса

**Как** авторизованный пользователь  
**Я хочу** открыть страницу конкретного сервиса  
**Чтобы** видеть основную информацию, историю версий и доступные модули

**Критерии приёмки**

- Страница сервиса построена по структуре `Overview` / `Settings`.
- В `Overview` есть блоки `Service information`, `Release`, `Versions history`, `Modules`.
- Основные данные берутся из PostgreSQL.
- Поле `version` отображает текущий git tag последнего релиза.
- Observer видит карточку сервиса и блок контрактов, но не получает доступ к логам, метрикам и действиям релиза.

## US-05. Просмотр контрактов сервиса

**Как** любой авторизованный пользователь  
**Я хочу** просматривать API-контракты сервиса  
**Чтобы** понимать, как интегрироваться с ним

**Критерии приёмки**

- MAMS получает `project.proto` из корня GitHub-репозитория сервиса.
- На странице сервиса есть вкладка `contracts`.
- Контракты отображаются в интерактивном виде со списком методов и параметров.
- Если `project.proto` отсутствует или невалиден, пользователь видит понятную ошибку.
- Вкладка доступна ролям `Developer`, `Service Owner`, `Observer`.

## US-06. Просмотр метрик через Grafana

**Как** пользователь с ролью `Developer` или `Service Owner` на сервисе  
**Я хочу** видеть метрики сервиса прямо в MAMS  
**Чтобы** не переключаться вручную между разными системами

**Критерии приёмки**

- На странице сервиса есть вкладка `metrics`.
- Во вкладке встраивается Grafana dashboard, связанный с сервисом по сохранённому идентификатору dashboard.
- Пользователь может просматривать метрики без перехода в отдельную админскую часть Grafana.
- Observer не видит вкладку `metrics`.

## US-07. Просмотр логов в реальном времени

**Как** пользователь с ролью `Developer` или `Service Owner` на сервисе  
**Я хочу** видеть логи сервиса в реальном времени  
**Чтобы** быстро разбирать проблемы после релиза и в ходе эксплуатации

**Критерии приёмки**

- На странице сервиса есть вкладка `logs`.
- Log Collector принимает поток логов из `stdin`.
- Новые логи сохраняются в MongoDB.
- Новые логи приходят в UI в реальном времени через WebSocket.
- В UI доступны фильтры по уровню, времени и тексту.
- Observer не видит вкладку `logs`.

## US-08. Обычный деплой сервиса

**Как** пользователь с ролью `Developer` или `Service Owner` на сервисе  
**Я хочу** запускать деплой сервиса из MAMS  
**Чтобы** не выполнять это вручную вне системы

**Критерии приёмки**

- В блоке `Release` можно выбрать стратегию `rolling` или `recreate`.
- Для `dev` и `staging` можно выбрать ветку.
- Для `prod` релиз опирается на git tag.
- MAMS запускает GitHub Actions workflow с параметрами деплоя.
- После запуска в истории релизов появляется новая запись со статусом.
- Observer не видит управляющие элементы релиза.

## US-09. Canary-деплой

**Как** пользователь с ролью `Developer` или `Service Owner` на сервисе  
**Я хочу** запускать canary-деплой  
**Чтобы** показать и проверить новую версию сервиса на ограниченном трафике

**Критерии приёмки**

- В блоке `Release` доступна стратегия `canary`.
- После запуска одновременно существуют stable и canary-версии сервиса.
- Traefik маршрутизирует запросы так, чтобы можно было отдельно открыть stable и canary-версию.
- Сценарий демонстрируется минимум на двух разных устройствах, браузерах или сессиях.
- История релизов фиксирует, что релиз был выполнен в стратегии `canary`.

## US-10. Rollback к предыдущей версии

**Как** пользователь с ролью `Developer` или `Service Owner` на сервисе  
**Я хочу** откатить сервис к предыдущему успешному релизу  
**Чтобы** быстро восстановить рабочее состояние

**Критерии приёмки**

- В `Release` есть режим rollback.
- При переходе в rollback обычные поля деплоя блокируются.
- Пользователь может выбрать предыдущий успешный git tag.
- После подтверждения MAMS запускает откат и создаёт запись в истории релизов.

## US-11. Настройки сервиса

**Как** пользователь с ролью `Developer` или `Service Owner` на сервисе  
**Я хочу** открыть настройки сервиса  
**Чтобы** видеть конфигурацию сервиса и управлять доступными мне правилами

**Критерии приёмки**

- На странице сервиса есть вкладка `Settings`.
- Вкладка доступна `Developer` и `Service Owner`.
- В настройках можно включить минимальный порог покрытия тестами.
- В настройках можно задать числовое значение порога покрытия.
- Настройка минимального порога покрытия является owner-only и редактируется только владельцем сервиса.
- Разработчик видит owner-only настройки в режиме чтения.
- Если порог включён и текущее покрытие ниже него, MAMS не разрешает запуск релиза.

## US-12. История релизов

**Как** авторизованный пользователь с доступом к сервису  
**Я хочу** видеть историю релизов  
**Чтобы** понимать, кто, когда и что выкатывал

**Критерии приёмки**

- В блоке `Versions history` отображается список релизов сервиса.
- Для каждой записи видны git tag, branch, статус, автор, дата и описание.
- Записи сортируются от новых к старым.
- После деплоя и rollback история обновляется без ручного вмешательства.

## US-13. Эффективная роль пользователя на сервисе

**Как** авторизованный пользователь  
**Я хочу** получать корректную роль на каждом сервисе  
**Чтобы** мои права зависели от конкретного сервиса, а не от одного глобального статуса

**Критерии приёмки**

- Роль пользователя определяется отдельно для каждого сервиса.
- Пользователь, создавший сервис, автоматически становится его `Service Owner`.
- Если пользователю явно выдан доступ уровня `Developer`, он получает соответствующие права на этом сервисе.
- Для всех остальных сервисов своей организации пользователь по умолчанию считается `Observer`.
- Один и тот же пользователь может быть `Service Owner` на одном сервисе, `Developer` на другом и `Observer` на третьем.

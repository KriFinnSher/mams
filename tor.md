# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## Система управления микросервисными приложениями (MAMS)

---

## 1. Цель

Разработать систему управления микросервисными приложениями и визуализации процессов их эксплуатации, которая объединяет в одном интерфейсе информацию о сервисе, релизы, логи, метрики и API-контракты.

---

## 2. Область применения и рамки первой версии

### Целевая аудитория

- команды разработки, работающие с несколькими связанными микросервисами;
- организации, которым нужен единый интерфейс для эксплуатации сервисов;
- разработчики, владельцы сервисов и наблюдатели.

### Что входит в первую версию

- login-форма и базовый профиль пользователя;
- добавление нового сервиса в систему через UI-форму;
- единая страница сервиса со структурой из презентации:
  - `Overview`;
  - `Settings`;
  - блок `Service information`;
  - блок `Release`;
  - блок `Versions history`;
  - блок `Modules` с вкладками `metrics`, `logs`, `contracts`;
- релизы и rollback;
- рабочий canary-деплой;
- просмотр логов, метрик и контрактов;
- базовые настройки сервиса.

### Что не входит в первую версию

- UI для создания пользователей и организаций;
- UI-админка для системного администрирования;
- `app.yaml` как источник истины для карточки сервиса.

> В первой версии источником истины для данных сервиса является PostgreSQL. Данные заносятся через форму добавления сервиса в MAMS. Поддержка импорта из `app.yaml` может быть добавлена позже.

## 3. Архитектура и развёртывание

### Общая схема

- **MAMS Core** разворачивается на отдельной виртуальной машине вне Kubernetes.
- **Kubernetes** используется как целевая среда запуска микросервисов.
- В системе используется один Kubernetes-кластер.
- Изоляция выполняется по схеме `organization + environment`.

### Правила изоляции

- Для каждой пары `организация + окружение` создаётся отдельный namespace.
- Рекомендуемый шаблон имени namespace: `<organization-slug>-<environment>`.
- Поддерживаемые окружения: `dev`, `staging`, `prod`.

### Репозитории и интеграция

- Один Git-репозиторий соответствует одному изолированному сервису.
- Конфигурационные файлы сервиса располагаются в корне репозитория.
- Файл `project.proto` обязателен для модуля контрактов.
- Для интеграции с GitHub в первой версии используется один технический токен, централизованно настроенный в MAMS, чтобы разработчики сервиса делали минимум дополнительных действий.

## 5. UX и визуальная структура

- Основной рабочий сценарий строится вокруг одной страницы сервиса.
- Структура интерфейса фиксируется по презентации: верхние вкладки `Overview` и `Settings`, далее блоки `Service information`, `Release`, `Versions history`, `Modules`.
- Визуальный стиль можно адаптировать под светлую тему, но композиция страницы и набор основных блоков должны сохраниться.
- Вкладка `Modules` должна отображать только те модули, которые доступны текущей роли.

## 6. Функциональные требования

### 6.1 Аутентификация и профиль

- В первой версии у пользователя есть только форма входа по выданным логину и паролю.
- После успешного входа пользователь получает JWT.
- В профиле пользователя отображаются:
  - login;
  - организация.
- Глобальная роль в профиле не отображается, так как права пользователя определяются в контексте конкретного сервиса.
- В профиле пользователя отображается список сервисов текущей организации, где его роль отличается от `Observer`, с указанием конкретной роли напротив каждого такого сервиса.

### 6.2 Добавление и хранение сервисов

- В системе должна быть отдельная страница добавления сервиса.
- Любой авторизованный пользователь может добавить новый сервис.
- После заполнения формы данные сохраняются в PostgreSQL.
- Именно данные из PostgreSQL используются для отображения карточки сервиса в интерфейсе.
- При создании сервиса пользователь, создавший его, автоматически становится владельцем сервиса.
- У сервиса в каждый момент времени может быть только один владелец.
- Для остальных пользователей той же организации новый сервис по умолчанию доступен с ролью `Observer`, пока им явно не выдан доступ уровня `Developer`.

### 6.3 Main Module

- Список сервисов организации.
- Страница конкретного сервиса с общей информацией.
- Поля сервиса:
  - `name` — название;
  - `description` — описание;
  - `type` — `business` | `composition`;
  - `version` — текущая версия, основанная на git tag последнего релиза;
  - `test_coverage` — тестовое покрытие в процентах;
  - `pii_sensitive` — работа с ПДн;
  - `responsible_team_ref` — ссылка или текстовый идентификатор команды, например `@infra-team`;
  - `importance` — категориальная важность (`low` | `medium` | `high` | `critical`);
  - `repository_url` — URL GitHub-репозитория;
  - `default_branch` — основная ветка сервиса;
  - `created_by_user_id` — пользователь, создавший сервис;
  - `owner_user_id` — назначенный владелец сервиса;
  - `grafana_dashboard_uid` — идентификатор Grafana dashboard, по которому MAMS формирует встроенное отображение.

### 6.4 Logs Module

- Просмотр логов в реальном времени.
- Фильтрация по уровню, времени и ключевым словам.
- Хранение логов в MongoDB.
- Передача логов в MAMS выполняется через Log Collector, который:
  - принимает поток логов из `os.stdin`;
  - сохраняет записи в MongoDB;
  - транслирует новые записи в UI через WebSocket.
- **Доступ**: `Developer`, назначенный `Service Owner`.

### 6.5 Metrics Module

- Отображение метрик через встроенный Grafana dashboard.
- Backend хранит идентификатор dashboard и отдаёт данные, необходимые для его встроенного отображения.
- Модуль предоставляет только чтение.
- **Доступ**: `Developer`, назначенный `Service Owner`.

### 6.6 Contract Module

- Визуализация API-контрактов.
- Источник данных — только `project.proto` в корне репозитория сервиса.
- Контракты отображаются в интерактивном виде с перечнем методов и параметров.
- **Доступ**: все авторизованные пользователи, включая `Observer`.

### 6.7 Release Module

- Поддерживаемые стратегии деплоя:
  - `rolling`;
  - `recreate`;
  - `canary`.
- Поддерживаемые окружения:
  - `dev`;
  - `staging`;
  - `prod`.
- Для `dev` и `staging` пользователь может выбрать ветку для деплоя.
- Для `prod` основной единицей версии является git tag.
- Git tag создаётся автоматически после merge в удалённый репозиторий по целевому процессу CI/CD.
- История релизов должна хранить и отображать:
  - стратегию;
  - окружение;
  - ветку;
  - git tag;
  - автора;
  - статус;
  - дату;
  - описание релиза.

### 6.8 Canary

- Canary должен быть реализован как реально работающий сценарий, пригодный для демонстрации.
- В системе одновременно существуют стабильная и canary-версии сервиса.
- Traefik должен уметь направлять запросы на stable или canary-версию так, чтобы можно было показать новую версию на одном устройстве и старую на другом.
- Для первой версии допускается упрощённая реализация маршрутизации через cookie, header или иной простой управляемый механизм.

### 6.9 Rollback

- Пользователь может выбрать одну из предыдущих успешных версий и откатить сервис к ней.
- Rollback выполняется по ранее задеплоенному git tag.
- Во время rollback поля обычного деплоя становятся неактивными, а вместо них появляется выбор целевой версии.

### 6.10 Settings

- Для сервиса должна существовать вкладка `Settings`.
- Вкладка `Settings` доступна ролям `Developer` и `Service Owner`.
- В первой версии в ней обязательно поддерживаются:
  - чекбокс включения минимального порога покрытия тестами;
  - числовое значение минимального порога покрытия.
- Настройка минимального порога покрытия относится к owner-only настройкам и изменяется только владельцем сервиса.
- Разработчик может открыть раздел `Settings` и просматривать owner-only настройки в режиме чтения.
- Если ограничение включено и текущее покрытие ниже порога, MAMS не должен разрешать запуск релиза.
- Раздел проектируется расширяемым для будущих настроек.

### 7.7 Связи

- одна организация содержит много пользователей;
- одна организация содержит много сервисов;
- один пользователь может создать много сервисов;
- один пользователь может быть владельцем нескольких сервисов;
- один пользователь может иметь разные роли на разных сервисах;
- у одного сервиса может быть только один назначенный владелец.
- у сервиса может быть несколько пользователей с ролью `Developer`.
- эффективная роль пользователя на сервисе определяется так:
  - если `user_id == owner_user_id`, то это `Service Owner`;
  - если у пользователя есть запись в `ServiceAccess` с ролью `developer`, то это `Developer`;
  - иначе это `Observer`.

## 8. Ролевая модель (RBAC)

Добавление нового сервиса доступно любому авторизованному пользователю. После создания сервиса его создатель автоматически становится `Service Owner` для этого сервиса.

| Функция | Developer | Service Owner | Observer |
|---------|-----------|---------------|----------|
| Вход в систему | ✓ | ✓ | ✓ |
| Просмотр сервисов | ✓ | ✓ | ✓ |
| Просмотр логов | ✓ | ✓ | ✗ |
| Просмотр метрик | ✓ | ✓ | ✗ |
| Просмотр контрактов | ✓ | ✓ | ✓ |
| Запуск деплоя | ✓ | ✓ | ✗ |
| Rollback | ✓ | ✓ | ✗ |
| Просмотр Settings | ✓ | ✓ | ✗ |
| Изменение owner-only настроек | ✗ | ✓ | ✗ |
| Просмотр профиля | ✓ | ✓ | ✓ |

> Роли `Developer`, `Service Owner` и `Observer` являются не глобальными, а эффективными ролями в контексте конкретного сервиса.

## 9. Интеграции

### 9.1 Kubernetes

- Интеграция через `github.com/kubernetes/client-go`.
- Namespace вычисляется по схеме `organization + environment`.
- Поддерживаемые операции:
  - deploy;
  - rollback;
  - status.

### 9.2 GitHub

- MAMS получает `project.proto` из GitHub-репозитория сервиса.
- MAMS инициирует GitHub Actions workflow для деплоя.
- GitHub используется как источник веток и git tag.
- В первой версии используется единый технический токен MAMS с достаточными правами на чтение репозитория и запуск workflow.

### 9.3 Grafana и Prometheus

- Grafana используется как встроенный UI для метрик.
- Prometheus используется как источник метрик для Grafana.
- MAMS хранит идентификатор Grafana dashboard и использует его для встроенного отображения метрик на странице сервиса.

### 9.4 Log Collector

- Отдельная сущность интеграции, принимающая поток логов из `stdin`.
- После приёма логов collector:
  - сохраняет их в MongoDB;
  - передаёт новые записи в MAMS для real-time отображения.

## 10. API Endpoints первой версии

### Auth

- `POST /api/auth/login` — вход в систему
- `GET /api/auth/me` — профиль текущего пользователя

### Services

- `GET /api/services` — список сервисов организации
- `POST /api/services` — добавление сервиса
- `GET /api/services/:id` — карточка сервиса
- `PUT /api/services/:id` — обновление общей информации о сервисе
- `PUT /api/services/:id/settings` — обновление настроек сервиса

### Logs

- `GET /api/services/:id/logs` — получение логов
- `GET /api/services/:id/logs/stream` — логи в реальном времени (WebSocket)

### Metrics

- `GET /api/services/:id/metrics` — данные для отображения Grafana dashboard

### Contracts

- `GET /api/services/:id/contracts` — получение API-контрактов сервиса

### Releases

- `GET /api/services/:id/releases` — история релизов
- `POST /api/services/:id/deploy` — запуск деплоя
- `POST /api/services/:id/rollback` — откат к предыдущему релизу

> CRUD пользователей и организаций не входит в первую UI-версию, но структура БД должна закладывать возможность добавить этот функционал позже.

## 11. CI/CD

### Основной процесс

- После merge в удалённый репозиторий создаётся git tag по принятой схеме версионирования.
- MAMS вызывает GitHub Actions workflow и передаёт параметры релиза.
- Workflow выполняет деплой в Kubernetes.

### Минимальные параметры workflow

```yaml
name: Deploy
on:
  workflow_dispatch:
    inputs:
      environment:
        required: true
        type: choice
        options: [dev, staging, prod]
      strategy:
        required: true
        type: choice
        options: [rolling, recreate, canary]
      branch:
        required: false
        type: string
      git_tag:
        required: false
        type: string
```

## 12. Критерии приёмки

1. Система разворачивается на виртуальной машине.
2. Работает форма входа и JWT-аутентификация.
3. Пользователь после входа видит только данные своей организации.
4. Добавление нового сервиса через UI-форму работает для любого авторизованного пользователя.
5. Создатель сервиса автоматически получает роль `Service Owner` для созданного сервиса.
6. Остальные пользователи организации по умолчанию получают доступ к этому сервису как `Observer`, если им не выдан уровень `Developer`.
7. В профиле пользователь видит список сервисов, где его роль отличается от `Observer`, и конкретную роль на каждом из них.
8. Карточка сервиса заполняется данными из PostgreSQL, а не из `app.yaml`.
9. У сервиса может быть только один назначенный владелец.
10. Наблюдатель видит контракты, но не видит логов, метрик и действий релиза.
11. Логи поступают в систему в реальном времени, сохраняются в MongoDB и отображаются в UI.
12. Метрики отображаются через встроенный Grafana dashboard.
13. Для привязки dashboard используется его идентификатор, а не вручную собранные графики внутри MAMS.
14. Контракты парсятся только из `project.proto`.
15. Деплой в Kubernetes работает для `rolling`, `recreate` и `canary`.
16. Canary можно продемонстрировать на двух разных устройствах или сессиях.
17. Rollback к предыдущему git tag работает.
18. История релизов отображает branch, git tag, статус, автора и дату.
19. Вкладка `Settings` доступна `Developer` и `Service Owner`, но owner-only настройки изменяет только владелец.
20. Настройка минимального порога покрытия блокирует релиз при нарушении условия.
21. Namespace разделяются по схеме `organization + environment`.
